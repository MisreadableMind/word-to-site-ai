ThemeREX WaaS Wizard

WordPress plugin providing a guided setup wizard for WaaS sites with AI-powered content generation and remote platform integration.


v1.0.0 WordPress Plugin PHP / PSR-4 REST API

Overview

ThemeREX WaaS Wizard is a WordPress plugin that automates the initial configuration of new WaaS (Website-as-a-Service) customer sites. It provides a 4-step guided admin wizard, AI-powered content and image generation, and a REST API for remote pushes from the WaaS platform.

The plugin self-deactivates automatically if required dependencies (TRX Addons and a ThemeREX theme with theme_pro_key) are not present.

Goals & Scope

In Scope

  • 4-step admin wizard UI with form handling
  • AI content generation for pages and blog posts
  • Remote configuration pushes from WaaS platform via REST
  • Theme/skin switching and demo data import
  • Image replacement via external image-bank service
  • OpenAI proxy registration for token-less AI access

Out of Scope

  • Front-end user-facing pages
  • Custom post types for content storage
  • Direct database queries (uses WP options / TRX Addons CPTs)

Key Design Patterns

Single-Option Data Store

All wizard state is persisted in one WP option: trx_waas_wizard_data. WizardDataStorage is the only class that reads/writes it. Never access the option directly.

Async / Cron Pattern

Long-running operations (generate-all, generate-images, switch-skin) are dispatched via wp_schedule_single_event() and polled by the caller through repeated REST requests.

Theme Integration

Never call theme functions directly. Use the Theme component wrapper:

Plugin::instance()->theme->func('function_name');
Plugin::instance()->theme->define('CONSTANT_NAME');

Hook Naming

All custom hooks use the trx_waas_wizard/ prefix:
Actions: trx_waas_wizard/loaded
Cron: trx_waas_wizard/rest_api/generate_all_content

Two-Layer Storage Architecture

Wizard flow state (step progress, completion flag, generated content lists) is stored in trx_waas_wizard_data and accessed exclusively via WizardDataStorage static methods.

Endpoint partition data (status strings, progress counters, runtime state) is stored in individual dedicated WP options, one per endpoint, managed by EndpointOptionStorage (includes/api/waas-platform/endpoint-option-storage.php). Keys are auto-derived: 'trx_waas_wizard_' + name.replace('-','_'). All reads/writes go through Base::get_part_data() / Base::update_part_data().

// Wizard flow option — trx_waas_wizard_data
{
  "current_step": 2,
  "completed":    false,
  "content_generation": { "pages": [], "posts_count": 5, ... }
}

// Per-endpoint options (examples)
// trx_waas_wizard_register_site
{ "inited": true, "status": "site_inited", "app_username": "...", "app_password": "..." }

// trx_waas_wizard_page_widgets
{ "status": "generate_all_end", "processed": 12, "skipped": 0, "errors": [] }

// trx_waas_wizard_save_wizard_data
{ "inited": true, "last_step": 3, "status": "all_steps_saved" }

Admin Wizard

The wizard appears as a menu item for admins after plugin activation. It guides through 4 steps and hides itself upon completion.

Step 1 — Site Settings

  • Site language (downloads language pack if needed)
  • Site title & tagline
  • Logo (file upload or URL)
  • Favicon (file upload, URL, or default plugin favicon)

Step 2 — User Profile

  • Username (stored, not changed)
  • Display nickname
  • Email address
  • Biographical info

Step 3 — Company Info

  • Name, industry, address, phone, email
  • Description, mission, history, values
  • Services list (cloneable: name, description, features)
  • Team members (cloneable: name, position, bio)
  • Saved to trx_addons_options + trx_waas_wizard_company_data

Step 4 — Content Generation

  • Select pages for AI rewrite
  • Set number of blog posts to generate
  • Trigger translation of page/post/layout titles
  • On submit: fires AJAX generation pipeline

AI Content Generation

AI generation uses TRX Addons AI Helper as the engine (must be enabled in TRX Addons options). Three generators are available:

PageContentGenerator

Rewrites existing Elementor section content in-place using AI. AJAX-driven (Step 4) or triggered via REST generate-section / generate-all.

BlogPostGenerator

Creates new blog posts from scratch using AI. Generates both the title and body. Triggered from Step 4.

NameTranslator

Translates page, post, and layout titles to the configured site language via batched AI calls. Phase 1 of the generate-all pipeline.

generate-all Async Status Machine

Status is stored in WP option trx_waas_wizard_page_widgets. Both generate-all and generate-images share this slot — they cannot run simultaneously.

Phase 2 dispatches one cron event per page (trx_waas_wizard/rest_api/generate_page_content), allowing pages to be processed in parallel. A pending_pages counter tracks completion; the last page job sets generate_all_end.

Image Pipeline

Phase 3 of content generation replaces images site-wide using the external content-bank.themerex.net service. The pipeline is designed for parallelism — all generation tasks are submitted before any polling begins.

ImageBankClient

Class: TrxWaasWizard\Includes\ImageBankClient — File: includes/image-bank-client.php

MethodHTTPEndpointNotes
search($path, $ctx)POST/api/searchmultipart/form-data; returns best match or null
generate($url, $ctx)POST/api/generateJSON body; returns task_id (int) or WP_Error
check_task($id)GET/api/tasks/{id}Single-shot, no loop — for round-robin use
poll_task($id)GET/api/tasks/{id}Blocking poll until success/error/timeout

Base URL is fixed: ImageBankClient::BASE_URL = 'https://content-bank.themerex.net'. All requests use HTTP Basic Auth.

Pipeline Steps

  1. build_image_task_map() — Deduplicates images by md5(url); collects all placements per unique URL.
  2. search_images() — Downloads each unique image temporarily, sends to /api/search; marks entries as found or need_generate based on score threshold.
  3. submit_generate_tasks() — Submits ALL generate tasks upfront (true parallelism on the server side).
  4. poll_and_import_images() — Imports already-found images; round-robin polls generating tasks via check_task().
  5. apply_elementor_updates() — Loads each post's Elementor data once, applies all replacements, saves once. Clears Elementor cache at the end.

Task Queue

A DB-backed async task queue that replaces the WP pseudo-cron per-page dispatch. Tasks run in true parallel via non-blocking HTTP requests.

Execution Flow

schedule_generate_all()
  1. DB::insert() × N tasks           → status = 'pending'
  2. update_part_data(group_id, ...)
  3. TaskQueue::dispatch([task_ids])   → N non-blocking HTTP POSTs
        ↓
POST /task-runner  (one per task, parallel)
  4. TaskQueue::claim(task_id)         → UPDATE WHERE status='pending' (atomic)
  5. do_action(hook, task_id, ...args)
        ↓
  schedule_generate_page(task_id, user_id, post_id, group_id)
  6. Process Elementor sections
  7. finish_page_job(task_id, ...)
        a. TaskQueue::complete(task_id)          → status = 'done'
        b. count_pending_in_group(group_id) → 0? → generate_all_end

Calling complete() before querying the group count is the key to avoiding a race condition — the current task is excluded from the count when the check runs.

Coding Conventions

  • Follow WordPress Coding Standards (WPCS).
  • Use plugin-defined constants (TRX_WAAS_WIZARD_*) instead of hardcoded paths/URLs.
  • Use the plugin's own helper/utility functions before writing new ones.
  • Nonce verification and capability checks are required for all AJAX/REST handlers.
  • No direct DB queries — use $wpdb with prepared statements or WP API wrappers.
  • Prefix all functions, hooks, and options with trx_waas_wizard_.
  • All code comments must be written in English.
  • Never access trx_waas_wizard_data directly — use WizardDataStorage.
  • Never call theme functions directly — use Plugin::instance()->theme->func().
