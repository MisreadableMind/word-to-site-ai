  • Website scraping (Firecrawl) Built
  • Deployment pipeline (InstaWP) Built
  • Domain + DNS + SSL (Namecheap + CF) Built
  • Light Editor (chat-based editing) Partial
  • User auth + sessions Partial

Schemas as Contracts

Onboarding and deployment are decoupled by two JSON schemas. The Wizard produces DeploymentContext (template, plugins, branding) and ContentContext (business info, pages, SEO). These merge into the WizardPayload sent to the client site's WAAS Controller.

Security Layer
Authentication Systems
Three separate auth mechanisms serving different purposes.

User Auth

Purpose: Platform login
Method: bcrypt + sessions + PostgreSQL
Scope: Dashboard, Wizard, Editor
Status: Partial
Planned: JWT + OAuth (Google, GitHub)

AI Proxy Keys (wts_)

Purpose: Client site → Platform AI calls
Method: Per-site API key prefixed wts_
Scope: AI Proxy plugin on WP site
Status: Planned
Why: Sites use AI without exposed keys

Plugin API Keys (ato_)

Purpose: Client site → Platform management API
Method: Per-site API key prefixed ato_
Scope: AI Traffic Optimizer plugin
Status: Planned
Why: Plugins report back to hub

Dev Mode Bypass

When NODE_ENV=development, basic auth is skipped entirely. This affects only User Auth — the AI Proxy and Plugin API keys will always be validated regardless of environment.
